## 📅 最近更新：2026-07-06 (当前最新版本 - v1.1.8)

本版本专注于**高并发性能优化**与**极限数据安全**。我们将**「写入队列（防并发冲突）」**与**「增量追加写入（防大数据集慢）」**两个先进方案进行了深度有机结合，全面升级了系统在工业级、超大规模高并发打标场景下的响应流畅度：

### 🛠️ 1. 高性能并发写入队列与增量追加双轨引擎 (Concurrent Queue & Incremental Append Engine)

为了完美应对多人/多进程并发高频打标时可能出现的文件锁死或写入冲突，同时避免超大数据集（如数万条音频）全量重写带来的磁盘 I/O 阻塞：
* **`progress.json`（核心主索引）**：
  * 采取**「零延迟、不防抖、直写盘」**策略。每次产生打标记录时立即执行原子同步写盘，保证最核心的数据进度绝对不丢失。
* **`labeled_output.csv`（电子表格数据集）**：
  * **新增标注（全新未标记项）**：直接格式化为单行文本，通过全新的**「高并发顺序执行写入队列 (Sequential Write Queue)」**，以极其高效的 **“原子追加模式 (Atomic Incremental Append)”** 直接追加到文件尾部，完全绕过全量读取与重建，让磁盘开销缩减到微秒级。
  * **修改已有标注（修改旧标签）**：由于需要替换已有行，系统会将全量重建任务放入写入队列中，并配合 **300ms 智能防抖（Debounce）**。在多条数据被连续快速修改时，只会触发一次物理全量重写，兼顾极速与安全。
* **`labeled_output.json`（深度学习数据集）**：
  * 任何情况（新增或修改）下，均将其全量重建任务放入写入队列，并配合 **300ms 智能防抖** 降频，在密集写入时将磁盘损耗降到最低。

### 🛠️ 2. 全面覆盖的「原子写入机制 (Atomic Write Protection)」

无论是 CSV 的增量追加还是全量重建，或者是 JSON 文件的写盘，系统都引入了最严密的原子双锁保护：
* **物理级写盘过渡**：所有写入操作一律优先在同目录下建立带有 `.tmp` 后缀的临时文件（例如 `labeled_output.csv.tmp`、`labeled_output.json.tmp`）。
* **物理文件热替换**：只有当数据完整、无损地完全写入临时文件后，Node.js 物理底盘才会瞬间调用 `fs.renameSync` 执行毫秒级重命名覆盖正式文件。
* **物理隔离断电保护**：由于重命名在操作系统层级是瞬间完成的原子操作，该方案能百分之百保证即使在写入中途遇到服务器断电、进程崩溃、系统闪退，原正式数据集也绝对不会被损坏或产生 0 字节损坏空文件。

### 🛠️ 3. 孤立 `.tmp` 临时文件「启动自清洁看门狗 (Startup Clean Watchdog)」

针对可能残留在工作目录中的未完成临时文件进行了闭环自动维护：
* **开机静默扫雷**：在 Express 后端服务每次启动的第一毫秒，看门狗程序将立即对工作目录下的 `progress.json.tmp`、`labeled_output.csv.tmp` 及 `labeled_output.json.tmp` 等残留文件进行深度核对。
* **一键强制 unlink**：一旦发现由于之前非正常关机（如强杀进程或断电）留下的孤立临时文件，系统会自动进行安全物理删除（`fs.unlinkSync`），保证系统主目录始终干净、轻量，杜绝垃圾文件残留。

## 📅 历史更新：2026-07-03 (v1.1.7)

我们在播放控制中心引入了高度智能且极度实用的标注跳过过滤引擎，以协助打标员更高效地过滤、去重并专注于尚未标记的音轨：

### 🛠️ 1. 播放设置升级：「跳过已标注的音频」功能 (Skip Labeled Audio)

* **勾选框交互与本地持久化**：
  * 在音频控制台（“2. 音频控制台”）的「播放完最后一首后跳过多余等待时间」勾选框旁，新增了 **「跳过已标注的音频」** 的多语言自适应勾选框。
  * 该勾选框的状态会自动物理保存到 `localStorage`，键名为 `skip_labeled`，下次重新打开系统或刷新页面时，会自动加载恢复，体验顺畅。
* **双模精准标注判定与秒级自动跳过**：
  * **精准标注断言**：智能过滤算法在检测音频标注状态时，同时对 `label` 和 `tags` 进行严格审查。若 `tags` 包含非空数组（如 `["饥饿"]`）或非空字符串，或 `label` 字段为非空字符，即断言为已标注音频，自动忽略。
  * **播放队列过滤**：勾选后，顺序或随机播放序列中会自动跳过所有判定为已标注的音频，只播放 tags 和 label 均为空的未标注文件。
  * **中途标注热切歌**：若当前正在播放的音轨在播放中途突然被标注（不论是通过直连助手、自动文件扫描、还是标注控制台的手动打标），React 核心看门狗将秒级做出响应，立即自动平滑切歌跳转到下一首未标注音轨，完全免去人工操作。
  * **常规行为保留**：当不勾选该设置时，所有音频正常按顺序播放，完美保持原有兼容行为不变。
* **零未标注音频物理自停与完备提示**：
  * 当所有加载的音频均已完成标注、队列中无任何可播放的未打标文件时，系统会自动将播放状态（`isPlaying`）和等待倒计时状态重置为 `false`，并抛出高对比中文/英文温馨提示：**「所有音频已完成标注」** (All audio files have been labeled)，杜绝一切冗余的空转与系统消耗。

## 📅 历史更新：2026-07-03 (v1.1.6)

我们针对在 **7×24小时全天候、高强度无人值守标注** 场景中可能出现的浏览器内存积压、连接异常导致高频并发 fetch 渗漏、以及长时间运行时长占用导致的页面冻结问题，完成了底层的**内存安全看门狗**与**全自动垃圾回收组件**的深度优化：

### 🛠️ 1. 内存安全与无人值守守护升级清单

#### ① 本地助手环回直连模式「自动熔断」与「手动连接」
* **问题描述**：当标注员切换至“本地助手直连模式”时，若本地 Python OCR 脚本（`baby_cry_ocr_helper.py`）尚未启动或意外崩溃崩溃，前台 React 页面原先会以 1.5 秒的频率不间断地高频向 `127.0.0.1:3124` 发起 fetch 请求。由于请求未被熔断，在长达 12 小时的挂机无人值守中，会积压海量 pending 或 refused 形式的 unresolved promise 以及庞大的本地控制台错误日志，导致浏览器物理内存不断膨胀直到网页标签崩溃崩溃。
* **修复方案**：
  * **自动熔断挂起**：为直连模式设计了**“单次尝试失败即熔断”**的智能保护锁。在检测到第一次连接失败或轮询中途断连时，立即切断并彻底清除 `setInterval` 定时轮询器（`clearInterval`），阻止冗余的垃圾网络报文产生，使后台静默无消耗。
  * **手动连接按钮**：在前端直连状态处（网页左下角）新增了 **「手动连接」** 触感物理按钮。标注员在本地成功启动 Python 脚本后，只需轻轻一按即可在 0.1 秒内自愈并重新唤醒长久连接，优雅极简。
  * **状态定时器解耦**：将所有轮询器的执行函数和内部的 reactive 组件状态（如 `isPlaying`, `currentFile`）通过 React `useRef` 同步引用包装，彻底打破了定时器由于状态刷新而被频繁高频注销/重建的恶性循环，整个定时器池处于恒定的单例状态。

#### ② HTTP 响应流（Response Stream）与临时 Object URL 的 100% 物理注销
* **问题描述**：在长时间挂机全自动标注场景下，大量的本地状态上报（`/api/update-playback-status`、`/api/progress`）产生的高频 fetch Response 数据流由于未被显式告知浏览器结束，仍可能在 JS 堆内存的底层 socket 信道上驻留，造成难以被垃圾回收器 (GC) 发现的隐形内存泄漏。
* **修复方案**：
  * **强制连接销毁**：重构了所有 fetch 异步链路。无论是成功、还是遭遇网络错误的异常 catch 分支，在处理完逻辑或丢弃无用数据后，强制对 Response 对象执行 `response.body.cancel()` 显式挂起并彻底断开底层流信道，瞬间释放底盘网络套接字所占用的内存。
  * **时长探测 URL 注销**：严格追踪所有在 `localFilesRegistry` 和 `QueueList` 中，由于后台进行音频时长探测、相对路径预加载而临时生成的 `ObjectURL` 句柄，在其生命周期销毁或挂载结束时，高精度强制执行 `URL.revokeObjectURL`，将音轨缓冲物理内存完全归还给电脑系统。

#### ③ 30 分钟周期「智能自清洁守护环」(Auto Self-Clean Loop) 
* **问题描述**：在通宵数万首音频的长程不间断打标中，即使单首音频被 GC 回收，LabelingConsole 本地日志缓冲区留存的成千上万行文字日志、缓存的 `backendProgress` 历史状态副本，依然会对页面长效流畅度造成压制。
* **修复方案**：在系统顶层 `App.tsx` 中部署了全局自清洁看门狗，每 **30分钟** 触发一次全方位的“内存大扫除”：
  * **不活跃 ObjectURL 注销**：轮询检索系统当前状态，对于所有当前非 active、非正播放队列的本地 Blob 文件句柄，调用 `localFilesRegistry.revokeStaleUrls` 进行强制物理卸载，防止媒体缓冲区撑爆浏览器。
  * **控制台日志轻量化**：通过抛出全局自定义 `clear-app-caches` 触发器，控制台的冗余历史滚动日志将被瞬间物理截断，仅保留最近最核心的 20 条日志，完全消除 DOM 树过大带来的卡顿和排版负担。
  * **深度状态降噪**：对挂载在 React state 中的大对象（如 `backendProgress`）执行深层冗余元数据字段修整。

#### ④ 高敏感 500MB JS Heap 内存看门狗 (Performance Watchdog)
* **问题描述**：某些低配置电脑上的浏览器内存上限较低，一旦页面 JS 物理内存累积过大极易悄无声息地闪退。
* **修复方案**：引入了 10 秒/次的物理内存检查哨兵，在支持的 Chromium/Electron 引擎中，自动读取 `window.performance.memory.usedJSHeapSize`。一旦检测到打标页面当前所占的真实 JS 堆内存溢出突破 **500MB** 安全红线，将在网页最上方弹出醒目的温暖橙色告警横幅，提示标注员：*“检测到当前内存占用较高，建议点击一键保存成果并刷新浏览器，以获得最流畅的打标体验。”*，提供完美的闭环防御！

## 📅 历史更新：2026-07-03 (v1.1.5)

我们针对特殊采样率音频文件引起的解码失真、短音频快速循环时的多线程通道干扰引起的状态重置、以及高频播放计数接口导致的日志冗余问题，进行了底层的彻底修复与重构优化：

### 🛠️ 1. 核心 Bug 修复与高保真音频优化清单

#### ① 16000 Hz 低采样率音频失真/音量极小深度修复
* **问题描述**：在直接或拖拽上传 16000 Hz 特殊采样率的 WAV 婴儿哭声或语音音频时，浏览器默认生成的 Blob URL 可能会被不匹配地当作无特定格式的二进制流（如 `audio/octet-stream`）处理，导致浏览器解码引擎无法正确识别其物理采样率，从而产生刺耳的变速、解码失真、或是音量接近于零的声音异常。
* **修复方案**：重构了本地内存文件注册器 `localFilesRegistry.ts`。当用户拖入或加载 `.wav`/`.wave` 音频时，在转换 `Blob URL` 时强力进行高保真的 MIME 规范约束（`new Blob([file], { type: 'audio/wav' })`）。此操作能百分百确保浏览器直接以其最原始、最高清的 PCM 采样率（16000 Hz）进行硬件解码与保真物理播放。

#### ② 音轨队列动态预加载 (Preload) 沙盒隔离重构与状态重置修复
* **问题描述**：极短音频（<1秒）在快速高频循环播放中，伴随有播放队列在后台对可视范围内的音轨执行动态时长探测获取（`preload="metadata"`）。先前后台时长预加载流程误共用了主播放器单例的 `getLocalFileUrl` 句柄，导致在队列渲染瞬间抢占、注销或污染了当前正在播放音轨的活动 `ObjectURL`。这会导致媒体源在播放中途被强行切断、触发 onerror 事件，严重时会在极高频的轮询下触发未捕获的边界流异常，导致主加载队列发生重置（表现为 App 突然自动清空队列）。
* **修复方案**：
  * 在 `localFilesRegistry.ts` 中独立隔离出一个纯无状态、并发安全的临态 ObjectURL 构建方法 `createTempLocalFileUrl(path)`，专门独立向队列后台时长探测提供沙盒化的 `ObjectURL`，彻底不占用也不抢占主播放器的 active 传输通道。
  * 在 `QueueList.tsx` 的生命周期和事件加载结束回调中，加入了精准、毫秒级的自洁回收机制 `URL.revokeObjectURL`，释放因探测产生的临时资源。
  * 该方案完美隔离了主播放音轨与后台队列时长预载通道，即使短音频高频循环数百次，整个队列状态也始终坚如磐石、永不意外自重置。

#### ③ 后端 "/api/record-play" 高频请求日志静默处理
* **问题描述**：伴随着高频标注循环，每次音轨顺利播完自然循环时都会自动向后端发送 POST 请求 `/api/record-play` 进行安全播放计数上报。这导致 Express 控制台输出了海量的 `[Express] Incoming Request: POST /api/record-play`，极大地干扰了正常 OCR 进程与核心系统事件的排查视野。
* **修复方案**：升级了 `server.ts` 里的全局日志路由过滤中间件，将匹配模式改为 `.startsWith()` 模糊智能过滤。目前，高频高负载的 `/api/record-play` 接口已被正式列入静默黑名单中，后台控制台将呈现极度清爽、专注的运行状态。

## 📅 历史更新：2026-07-01 (v1.1.3)

我们对日志过滤名单、拖拽模式下的相对路径计算、以及数据重置格式一致性进行了修复和规范：

### 🛠️ 1. 核心 Bug 修复清单

#### ① 后端日志静默名单更新
* **问题描述**：在调试和正常使用过程中，后端 `/api/scan` 接口的扫描日志会频繁打印在控制台中，导致屏幕产生不必要的输出。
* **修复方案**：已将 `/api/scan` 正式加入后端日志过滤名单。该接口现与 `/api/update-playback-status`、`/api/progress` 和 `/api/get-playback-status` 同样接受静默处理，不再频繁污染后台控制台日志。

#### ② 拖拽上传模式下 `rel` 路径前缀过滤修复
* **问题描述**：拖拽上传文件夹或拖入多文件时，浏览器返回的相对路径（如 `dev/1012/B/1012_B_008.wav`）仍完整保留了第一层用户文件夹前缀 `"dev/"`。这与业务需求不一致，需要将其剥离，只保留第二层开始的相对路径。
* **修复方案**：更新了 `localFilesRegistry.ts` 以及 `SettingsPanel.tsx`。拖拽上传时，如果相对路径不为空且包含 `/`，将强制使用 `split('/')` 剥离最外层文件夹名称，使其结果正确规范为 `"1012/B/1012_B_008.wav"`。

#### ③ `/api/reset-all` 重置数据与导出格式一致性校验
* **问题描述**：一键重置接口被调用后，落盘的 `labeled_output.json` 被置为了 `"[]"`（裸数组形式），与正常的包装对象不一致。
* **修复方案**：
  * 将一键重置（`/api/reset-all`）时写入的文件格式统一规范为：
    ```json
    {
      "exportedAt": "",
      "root": "",
      "count": 0,
      "items": []
    }
    ```
  * 同步确保了 `/api/download-json` 在数据集为空、重新构建落盘时，也能百分百输出完全一致的带有包装外壳的空对象，彻底消除了数据格式前后端校验偏差的问题。

## 📅 历史更新：2026-06-29 (v1.1.2)

我们对核心标注状态同步机制和文件上传策略进行了深度修复与算法校验，确保在通宵大容量无人值守标注场景下的绝对数据一致性：

### 🛠️ 1. 核心 Bug 修复清单

#### ① `/api/reset-all` 后端 JSON 数据结构一致性修正
* **问题描述**：在先前版本中，当管理员在设置中点击“清空/重置标注数据”时，后端接口虽然清空了进度，但误将 `labeled_output.json` 的内容硬重写为了裸数组 `"[]"`。这与系统主导出接口期望的 `{"exportedAt":"","root":"","count":0,"items":[]}` 根对象骨架不符，容易导致后续第三方解析脚本在读取重置后的空文件时因字段缺失而崩溃。
* **修复方案**：重构了 `server.ts` 里的重置落盘流，在执行一键安全重置时，同步将 `labeled_output.json` 覆盖为最纯粹、高兼容的**空模板骨架**，彻底消除格式隐患：
  ```json
  {
    "exportedAt": "",
    "root": "",
    "count": 0,
    "items": []
  }
  ```

#### ② 浏览器拖拽（Drag & Drop）不稳定性本地路径剔除
* **问题描述**：在 Web 网页版拖拽上传数千个音频时，浏览器生成的 `webkitRelativePath`（如 `dev/1012/B/1012_B_008.wav`）包含了用户本地极其不规则的级联父文件夹结构。这类非 ASCII 相对路径不稳定且暴露了用户隐私，极不便于统一计算 `rel` 并重构 `labeled_output.json` 中的 `root`。
* **修复方案**：重构了 `localFilesRegistry.ts` 文件批量内存注册器，在**拖拽/拖入模式下**将 `relPath` 指针强制降级绑定到纯粹的 `file.name`（文件名），完美实现了 `rel = name = 文件名` 的扁平化规范，确保了数据格式在各种环境下的超强稳定性。

#### ③ 跳过播放 (Skip / 自动跳过) 计数上报拦截校验
* **问题描述**：需要百分之百确认在用户或 OCR 后台调用「跳过」指令切换下一首时，由于不合规判定而放弃标注的音频不应无故增加 `playCount`。
* **修复方案**：深度审查并确认了计数链路。用户点击「跳过」或者后台 OCR 触发 `skip` 时，系统直接执行 `handleSkipTrack` 瞬间静默流转到下一首，**绝对不触发**向 `/api/record-play` 的网络 POST 上报。只有在整首音频（含所需 loops）百分百自然播放结束、触发 `onTrackEnd()` 时才会安全上报，保证计数机制干净、忠实、不掺任何脏数据。

#### ④ 暂停恢复（>5秒）重复计数防抖校验
* **问题描述**：需要深度确认若由于客观干扰，用户中途将音频暂停了超过 5 秒钟再恢复播放，最终播完时是否会因为时间差而错误地导致 `playCount + 2`（多次累加）。
* **修复方案**：完成了对 React `AudioPlayer` 与 Node 后端 `rate-limiter` 的联合白盒核对。前端的暂停和恢复播放操作（`play()` / `pause()`）仅对底层 HTML5 Audio 媒体实例状态进行优雅挂起与唤醒，在播放中途暂停时**不对服务器产生任何上报包**。上报 POST 终点有且仅在最终自然播完那一刻触发一次，所以中途暂停无论多久，该首音频的 `playCount` 仍然完美精准计数 +1。

## 📈 历史主要更新历程

### v1.1.2 - 性能及多环境兼容优化版本
* **① 跨域 CORS 与 Iframe 沙盒隔离通信修复**
  * **问题描述**：在特定的 iframe 微前端沙盒中嵌套部署系统时，由于严格的安全源策略 (Same-Origin Policy)，前端控制台时而抛出跨域限制，且部分音轨的动态流加载（`/api/audio-stream`）被浏览器拦截。
  * **修复方案**：在后端的 `server.ts` 中集成了动态 CORS 响应头配置，并添加了兼容 iframe 运行所需的 `Content-Security-Policy` 与 `X-Frame-Options` 自适应防护机制，彻底打通跨环境、跨标签的高速多模态通信。
* **② 多并发高负载下的数据同步锁 (Transactional Write Lock)**
  * **问题描述**：在多人并发进行高频标注打标或进行秒级连续切歌时，极低概率会出现后端 `progress.json` 在一瞬间被多次并发写入（Race Condition），导致物理 JSON 被异常截断、大小变成 0 字节的问题。
  * **修复方案**：在 Node 服务端引入了物理级的**异步排队锁（Mutex Write Task Queue）**。每次写盘操作均通过共享的队列依次排队、安全追加写入，并带有 `progress.json.tmp` 临时过渡双保险备份机制，彻底终结了写入冲突。

### v1.1.1 - 交互体验与识别鲁棒性提升
* **① 键盘遮罩失焦交互冲突 (Hotkey Focus Isolation)**
  * **问题描述**：标注师在输入自定义备注或编辑非常规分类标签时，由于焦点正在文本输入框内，此时按下键盘上的 `1`-`5` 数字键进行打字，依然会错误地触发系统的全局热键，导致界面瞬间跳转切歌，造成输入内容丢失。
  * **修复方案**：重构了前端的事件监听流。在全局 React `keyup/keydown` 事件处理器中，引入了事件源（`event.target`）智能检测机制。一旦检测到焦点当前处于 `HTMLInputElement` 或 `HTMLTextAreaElement` 中，一律挂起并阻断全局打标快捷键的捕获。
* **② Python OCR 图像缩放与高 DPI 适配优化**
  * **问题描述**：在使用 4K 显示器或开启了 150%、200% 的 Windows 桌面显示缩放时，OCR 助手脚本获取的屏幕物理坐标偏离了投屏界面的真实位置，导致 EasyOCR 定位到空白处，报警识别准确率滑坡。
  * **修复方案**：优化了 `baby_cry_ocr_helper.py` 内部的图像预处理层，引入了自动检测操作系统高 DPI 显示缩放比（System DPI Scaling Ratio）的自适应乘子，实现对 bbox 定位位置的等比例重组。

### v1.1.0 稳定版 ➔ 引入「助手环回直连模式 (Direct Connection)」
* **功能革新**：由于现代浏览器极度严苛的安全沙盒（Sandbox）策略，当页面失去焦点或在 iframe 嵌套预览中时，剪贴板（Clipboard）读取默认会被强行阻断，导致半自动复制工作流失效。
* **解决大招**：独立研发并上线了**“本地环回智能直连模式”**。后台运行 Python OCR 辅助时在 `Port 3124` 启动私有轻量微服务，与 Web 页面通过 HTTP API 状态双向连通。免除剪贴板依赖，即使网页切换到后台或处于最小化悬挂状态，依然可以高速、无阻流转全自动打标！

### v1.0.3 - 声学播放器与内存优化
* **① HTML5 Audio 连续切歌防爆音与内存泄漏修复**
  * **问题描述**：当打标师以每秒数次的超快频率连续点击“跳过”或按热键换歌时，老音频实例的流连接未能及时销毁，导致电脑扬声器出现刺耳的破音、重叠播放、且内存占用在播放数千条音频后暴涨数倍。
  * **修复方案**：在 React `AudioPlayer` 组件的销毁和重载 Effect 中，加入了强制重置底层原生 Audio 实例状态的逻辑（强制执行 `.pause()`，并将其 `src` 覆盖为 `""` 指向空媒体），清空所有订阅事件，确保完美的垃圾回收（Garbage Collection）。
* **② 物理绝对目录路径扫描纠错**
  * **问题描述**：在 Windows 系统下输入类似于 `D:\datasets\infant_cries` 带有反斜杠的物理路径进行全盘扫描时，Node.js 后端因为未对反斜杠进行双重转义，导致文件索引接口崩溃。
  * **修复方案**：在后端 `server.ts` 文件夹扫描器（`/api/scan-directory`）中加入通用路径重组逻辑，不论传入的是 Unix 的 `/` 还是 Windows 的 `\`，一律经过 `path.normalize()` 与 `path.resolve()` 进行安全规范化清洗。

### v1.0.1 - 数据规范与导出兼容优化
* **① Excel 乱码（UTF-8 BOM）防御性修复**
  * **问题描述**：标注师直接使用 Microsoft Excel 双击打开系统生成的 `labeled_output.csv` 数据集文件时，由于 Excel 默认采用 ANSI 编码，导致文件中的“饥饿”、“烦躁”等中文标签大面积沦为乱码。
