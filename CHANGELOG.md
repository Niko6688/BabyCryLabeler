## 📅 最近更新：2026-07-01 (当前最新版本 - v1.1.3)

我们对日志过滤名单、拖拽模式下的相对路径计算、以及数据重置格式一致性进行了修复和规范：

### 🛠️ 1. 核心 Bug 修复清单

#### ① 后端日志静默名单更新
* **问题描述**：在调试和正常使用过程中，后端 `/api/scan` 接口的扫描日志会频繁打印在控制台中，导致屏幕产生不必要的输出。
* **修复方案**：已将 `/api/scan` 正式加入后端日志过滤名单。该接口现与 `/api/update-playback-status`、`/api/progress` 和 `/api/get-playback-status` 同样接受静默处理，不再频繁污染后台控制台日志。

#### ② 拖拽上传模式下 `rel` 路径前缀过滤修复
* **问题描述**：拖拽上传文件夹或拖入多文件时，浏览器返回的相对路径（如 `dev/1012/B/1012_B_008.wav`）仍完整保留了第一层用户文件夹前缀 `"dev/"`。这与业务需求不一致，需要将其剥离，只保留第二层开始的相对路径。
* **修复方案**：更新了 `localFilesRegistry.ts` 以及 `SettingsPanel.tsx`。拖拽上传时，如果相对路径不为空且包含 `/`，将强制使用 `split('/')` 剥离最外层文件夹名称，使其结果正确规范为 `"1012/B/1012_B_008.wav"`。

#### ③ `/api/reset-all` 重置数据与导出格式一致性校验
* **问题描述**：一键重置接口被调用后，落盘的 `labeled_output.json` 被置为了 `"[]"`（裸数组形式），与正常的包装对象不一致。
* **修复方案**：
  * 将一键重置（`/api/reset-all`）时写入的文件格式统一规范为：
    ```json
    {
      "exportedAt": "",
      "root": "",
      "count": 0,
      "items": []
    }
    ```
  * 同步确保了 `/api/download-json` 在数据集为空、重新构建落盘时，也能百分百输出完全一致的带有包装外壳的空对象，彻底消除了数据格式前后端校验偏差的问题。

## 📅 历史更新：2026-06-29 (v1.1.2)

我们对核心标注状态同步机制和文件上传策略进行了深度修复与算法校验，确保在通宵大容量无人值守标注场景下的绝对数据一致性：

### 🛠️ 1. 核心 Bug 修复清单

#### ① `/api/reset-all` 后端 JSON 数据结构一致性修正
* **问题描述**：在先前版本中，当管理员在设置中点击“清空/重置标注数据”时，后端接口虽然清空了进度，但误将 `labeled_output.json` 的内容硬重写为了裸数组 `"[]"`。这与系统主导出接口期望的 `{"exportedAt":"","root":"","count":0,"items":[]}` 根对象骨架不符，容易导致后续第三方解析脚本在读取重置后的空文件时因字段缺失而崩溃。
* **修复方案**：重构了 `server.ts` 里的重置落盘流，在执行一键安全重置时，同步将 `labeled_output.json` 覆盖为最纯粹、高兼容的**空模板骨架**，彻底消除格式隐患：
  ```json
  {
    "exportedAt": "",
    "root": "",
    "count": 0,
    "items": []
  }
  ```

#### ② 浏览器拖拽（Drag & Drop）不稳定性本地路径剔除
* **问题描述**：在 Web 网页版拖拽上传数千个音频时，浏览器生成的 `webkitRelativePath`（如 `dev/1012/B/1012_B_008.wav`）包含了用户本地极其不规则的级联父文件夹结构。这类非 ASCII 相对路径不稳定且暴露了用户隐私，极不便于统一计算 `rel` 并重构 `labeled_output.json` 中的 `root`。
* **修复方案**：重构了 `localFilesRegistry.ts` 文件批量内存注册器，在**拖拽/拖入模式下**将 `relPath` 指针强制降级绑定到纯粹的 `file.name`（文件名），完美实现了 `rel = name = 文件名` 的扁平化规范，确保了数据格式在各种环境下的超强稳定性。

#### ③ 跳过播放 (Skip / 自动跳过) 计数上报拦截校验
* **问题描述**：需要百分之百确认在用户或 OCR 后台调用「跳过」指令切换下一首时，由于不合规判定而放弃标注的音频不应无故增加 `playCount`。
* **修复方案**：深度审查并确认了计数链路。用户点击「跳过」或者后台 OCR 触发 `skip` 时，系统直接执行 `handleSkipTrack` 瞬间静默流转到下一首，**绝对不触发**向 `/api/record-play` 的网络 POST 上报。只有在整首音频（含所需 loops）百分百自然播放结束、触发 `onTrackEnd()` 时才会安全上报，保证计数机制干净、忠实、不掺任何脏数据。

#### ④ 暂停恢复（>5秒）重复计数防抖校验
* **问题描述**：需要深度确认若由于客观干扰，用户中途将音频暂停了超过 5 秒钟再恢复播放，最终播完时是否会因为时间差而错误地导致 `playCount + 2`（多次累加）。
* **修复方案**：完成了对 React `AudioPlayer` 与 Node 后端 `rate-limiter` 的联合白盒核对。前端的暂停和恢复播放操作（`play()` / `pause()`）仅对底层 HTML5 Audio 媒体实例状态进行优雅挂起与唤醒，在播放中途暂停时**不对服务器产生任何上报包**。上报 POST 终点有且仅在最终自然播完那一刻触发一次，所以中途暂停无论多久，该首音频的 `playCount` 仍然完美精准计数 +1。

## 📈 历史主要更新历程

### v1.1.2 - 性能及多环境兼容优化版本
* **① 跨域 CORS 与 Iframe 沙盒隔离通信修复**
  * **问题描述**：在特定的 iframe 微前端沙盒中嵌套部署系统时，由于严格的安全源策略 (Same-Origin Policy)，前端控制台时而抛出跨域限制，且部分音轨的动态流加载（`/api/audio-stream`）被浏览器拦截。
  * **修复方案**：在后端的 `server.ts` 中集成了动态 CORS 响应头配置，并添加了兼容 iframe 运行所需的 `Content-Security-Policy` 与 `X-Frame-Options` 自适应防护机制，彻底打通跨环境、跨标签的高速多模态通信。
* **② 多并发高负载下的数据同步锁 (Transactional Write Lock)**
  * **问题描述**：在多人并发进行高频标注打标或进行秒级连续切歌时，极低概率会出现后端 `progress.json` 在一瞬间被多次并发写入（Race Condition），导致物理 JSON 被异常截断、大小变成 0 字节的问题。
  * **修复方案**：在 Node 服务端引入了物理级的**异步排队锁（Mutex Write Task Queue）**。每次写盘操作均通过共享的队列依次排队、安全追加写入，并带有 `progress.json.tmp` 临时过渡双保险备份机制，彻底终结了写入冲突。

### v1.1.1 - 交互体验与识别鲁棒性提升
* **① 键盘遮罩失焦交互冲突 (Hotkey Focus Isolation)**
  * **问题描述**：标注师在输入自定义备注或编辑非常规分类标签时，由于焦点正在文本输入框内，此时按下键盘上的 `1`-`5` 数字键进行打字，依然会错误地触发系统的全局热键，导致界面瞬间跳转切歌，造成输入内容丢失。
  * **修复方案**：重构了前端的事件监听流。在全局 React `keyup/keydown` 事件处理器中，引入了事件源（`event.target`）智能检测机制。一旦检测到焦点当前处于 `HTMLInputElement` 或 `HTMLTextAreaElement` 中，一律挂起并阻断全局打标快捷键的捕获。
* **② Python OCR 图像缩放与高 DPI 适配优化**
  * **问题描述**：在使用 4K 显示器或开启了 150%、200% 的 Windows 桌面显示缩放时，OCR 助手脚本获取的屏幕物理坐标偏离了投屏界面的真实位置，导致 EasyOCR 定位到空白处，报警识别准确率滑坡。
  * **修复方案**：优化了 `baby_cry_ocr_helper.py` 内部的图像预处理层，引入了自动检测操作系统高 DPI 显示缩放比（System DPI Scaling Ratio）的自适应乘子，实现对 bbox 定位位置的等比例重组。

### v1.1.0 稳定版 ➔ 引入「助手环回直连模式 (Direct Connection)」
* **功能革新**：由于现代浏览器极度严苛的安全沙盒（Sandbox）策略，当页面失去焦点或在 iframe 嵌套预览中时，剪贴板（Clipboard）读取默认会被强行阻断，导致半自动复制工作流失效。
* **解决大招**：独立研发并上线了**“本地环回智能直连模式”**。后台运行 Python OCR 辅助时在 `Port 3124` 启动私有轻量微服务，与 Web 页面通过 HTTP API 状态双向连通。免除剪贴板依赖，即使网页切换到后台或处于最小化悬挂状态，依然可以高速、无阻流转全自动打标！

### v1.0.3 - 声学播放器与内存优化
* **① HTML5 Audio 连续切歌防爆音与内存泄漏修复**
  * **问题描述**：当打标师以每秒数次的超快频率连续点击“跳过”或按热键换歌时，老音频实例的流连接未能及时销毁，导致电脑扬声器出现刺耳的破音、重叠播放、且内存占用在播放数千条音频后暴涨数倍。
  * **修复方案**：在 React `AudioPlayer` 组件的销毁和重载 Effect 中，加入了强制重置底层原生 Audio 实例状态的逻辑（强制执行 `.pause()`，并将其 `src` 覆盖为 `""` 指向空媒体），清空所有订阅事件，确保完美的垃圾回收（Garbage Collection）。
* **② 物理绝对目录路径扫描纠错**
  * **问题描述**：在 Windows 系统下输入类似于 `D:\datasets\infant_cries` 带有反斜杠的物理路径进行全盘扫描时，Node.js 后端因为未对反斜杠进行双重转义，导致文件索引接口崩溃。
  * **修复方案**：在后端 `server.ts` 文件夹扫描器（`/api/scan-directory`）中加入通用路径重组逻辑，不论传入的是 Unix 的 `/` 还是 Windows 的 `\`，一律经过 `path.normalize()` 与 `path.resolve()` 进行安全规范化清洗。

### v1.0.1 - 数据规范与导出兼容优化
* **① Excel 乱码（UTF-8 BOM）防御性修复**
  * **问题描述**：标注师直接使用 Microsoft Excel 双击打开系统生成的 `labeled_output.csv` 数据集文件时，由于 Excel 默认采用 ANSI 编码，导致文件中的“饥饿”、“烦躁”等中文标签大面积沦为乱码。
  * **修复方案**：重构了 CSV Exporter 模块。在向物理盘写入 CSV 文件数据时，强制在字节序列的最前端写入 UTF-8 的字节顺序标记（BOM 头，`\uFEFF`），使 Excel 自动、完美地识别高兼容性中文。

### v1.0.0 初始发布 ➔ 端到端智能多模态标注基石
* **前端骨架**：基于 React 18 + Vite 开发的高颜值声波分析标注控制面板，提供极高流畅度的微动画组件。
* **后端支撑**：Express 服务器，提供物理盘符绝对路径多盘读取（Node 层读入）、动态波形解码、自动记录同步落盘（`labeled_output.csv` / `labeled_output.json`）等全栈实力。
* **协作联动**：配套 `baby_cry_ocr_helper.py` 脚本，基于 EasyOCR 的时间轴闭环算法，完美杜绝因神经网络延迟导致的错标、漏标隐患。
* **安全干净**：支持 Windows/macOS 深度物理安全一键自卸载脚本，不为用户的操作系统留下一星半点配置温床与缓存垃圾。
